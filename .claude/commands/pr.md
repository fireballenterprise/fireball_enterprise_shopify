# Create Pull Request into `development`

## Step 1 — Confirm you are on a feature branch

Run:
```
git rev-parse --abbrev-ref HEAD
```

If the current branch is `main`, `development`, or `wip_deploy_01`, **stop** and tell the user they must switch to a feature branch before opening a PR.

## Step 2 — Gather the diff against `development`

Gather the branch and diff context by running `uv run --no-sync invoke repo.pr_diff` using the Bash
tool. If it fails, show the full output to the user and ask how they'd like to proceed.

Use this output to understand what changed. Do **not** print the full raw diff to the user.

## Step 3 — Check for uncommitted or unpushed changes

1. Run `git status --short` — if there are uncommitted changes, ask the user if they want to commit them first.
2. Run `git status -sb` to check if the branch is ahead of its remote. If unpushed commits exist, run `git push --set-upstream origin HEAD` before creating the PR.

## Step 4 — Write the PR title and description

**Title**: Imperative mood, under 72 characters, describing *what* the PR does.

**Body** (use this structure exactly):

```
## Summary

- <bullet: key change or feature>
- <bullet: another key change>
- <bullet: any notable implementation decision or dependency>

## Details

<2–4 sentence narrative of what changed and why, written for a reviewer.>

## Changes

<auto-generated from `git diff --stat` output — list the files/areas touched>
```

Populate all three sections from the diff data gathered in Step 2.

## Step 5 — Create the PR

This repository is a GitHub **fork** of `Shopify/dawn`. Tools and `gh` default the base
repository to the upstream parent (`Shopify/dawn`) unless told otherwise — always pass the
repo explicitly so the PR is opened against the fork, not upstream.

Run:
`uv run --no-sync invoke repo.pr_create --title="<title>" --content="<notes>"`

This targets `fireballenterprise/shopify_dawn_theme` explicitly (not upstream) and bases the PR
on `development`. If a PR already exists for this branch, it reports its URL instead of erroring —
just relay that to the user. Report the PR URL to the user.
